대시보드를 처음 접한 신규 담당자를 위한 5분 가이드입니다. 본 매뉴얼 전체를 읽기 전에 우선 이 페이지로 핵심 작업 흐름을 이해하실 수 있습니다.

당신이 해야 할 일

효성티앤씨 ESG경영팀의 환경 정량 데이터(50개 지표 × 18개 사업장 × 3개년)를 매년 수집·검수·발행하는 업무입니다. 본 대시보드는 그 작업의 단일 진실 원천(SSOT)입니다.

Day 1에 반드시 알아야 할 4가지

1.3 보고서 작성 프로세스 + 외부 검증 흐름

지속가능경영보고서 환경 정량 데이터 작성은 다음 5단계 프로세스를 거칩니다.

전체 프로세스

  1. 1단계 (1~3월): 데이터 수집
    • 본사 ESG경영팀이 18개 사업장에 본사 제공 템플릿 발송
    • 각 법인이 환경 정량 데이터 (57개 지표 × 3개년) 회신
    • 본사가 SAP 재고수불부·G/L에서 HQ_MANAGED 11개 지표 자체 산출
  2. 2단계 (3~4월): 데이터 검수
    • 법인 회신 엑셀 7단계 점검 절차 (§5.1)
    • YoY 이상치 검출 + 법인 추가 질의
    • RAW·V·NA 등록 표준 절차 (§6)
  3. 3단계 (4월 말): 본사 일괄 산출
    • 재고수불부 기반 11개 HQ_MANAGED 지표 산출 (행85·86·87·88·89·90·91·92·93·103·112)
    • 매칭 엔진 v2 (R103) + R090 v4 적용
  4. 4단계 (5월 초): 외부 검증사 송부
    • 대시보드 + 외부 심사원 엑셀 + 증빙 자료 송부
    • 11개 시트 정합성 점검 후 발송
  5. 5단계 (5월 중·말~6월 초): 외부 검증 의견 수령 + 정정 + 보고서 발행
    • 외부 검증 의견 수령
    • 지적 사항 답변 + 정정 작업
    • 6월 초 지속가능경영보고서 발행

외부 검증 흐름

외부 검증사는 GRI Standards + ISAE 3000 기준에 따라 다음 순서로 검증을 진행할 수 있습니다.

  1. 대시보드 + 외부 심사원 엑셀 수령 → 11개 시트 + 50개 지표 ↔ 매칭 확인
  2. RAW 데이터 ↔ 증빙 자료 교차 검증 → 법인 제출값 ↔ 정부 공식 문서 ↔ SAP 자체 산출 일관성 점검
  3. V/NA basis 텍스트 검토 → 다국어 ko/en/zh 일관성 + 산출 정의 명확성
  4. 이상치·갭 항목 질의 → YoY 이상치·일부 미등록·법인 회신 차이 답변 요구
  5. 의견서 발급 → 한정의견 또는 적정의견

외부 검증사 예상 질의 유형

  • RAW 값의 출처 (정부 공식 문서·법인 자체 측정·SAP 산출 중 어느 것인가?)
  • YoY 이상치 사유 (예: 닝샤 행75 용수 재활용 269,116% 폭증 → C11 신증설)
  • NA 등록 사유 (구조적 부재·사업 유형 불일치 등 명확한 근거)
  • 다국어 답변 일관성 (ko·en·zh 동일 정보 여부)
  • HQ_MANAGED 산출 절차 (재고수불부 ↔ 매칭 엔진 ↔ GHS 마스터 매핑)

1.4 ESG경영팀 담당자 역할 + 책임 범위

대시보드 담당자는 다음 5종 책임을 가집니다.

역할 1: 데이터 검수자

  • 법인 회신 데이터 정확성 확인 (7단계 점검 절차)
  • YoY 이상치·일부 미등록 검출 + 추가 질의
  • 본사 일괄 산출 11개 지표 (재고수불부·SAP 기반) 산출

3장. 핵심 용어 사전

3.1 RAW vs VERIFIED vs NOT_APPLICABLE 차이

대시보드 6객체 중 데이터 핵심 3종 객체의 역할 구분입니다.

RAW

  • 역할: 각 사업장의 측정값 원본 데이터
  • 구조: { "사업장명": [FY2023값, FY2024값, FY2025값] }
  • 값 유형: 숫자 (실측값) 또는 null (미입력) 또는 0 (실적 없음)
  • 변경 빈도: 법인 회신 또는 정정 시
  • 위험도: 정정 시 위험도 A (백업·검증 필수)

VERIFIED (V)

  • 역할: 본사 검수 완료 후 등록한 검증 자료 (다국어 ko/en/zh)
  • 구조: { "사업장명:지표명": { ko: {confirmed, basis, meta}, en: {...}, zh: {...} } }
  • 키 형식: "사업장명:지표명" (콜론으로 결합)
  • 변경 빈도: 신규 등록 또는 basis 갱신 시
  • 위험도: 신규 등록·basis 갱신 시 위험도 A

NOT_APPLICABLE (NA)

  • 역할: 구조적으로 해당 없음 (시설 부재·사업 유형 불일치 등)
  • 구조: V와 동일 ("사업장명:지표명": { ko, en, zh })
  • RAW 값: null 또는 0 (NA 등록 시 RAW 정합성 점검 필수)
  • 변경 빈도: 신규 등록 또는 사유 갱신 시
  • 위험도: 신규 등록 시 위험도 A

핵심 원칙

V·NA 동일 키 동시 등록 금지 (충돌 방지). V 등록 후 NA 변경 시 V 삭제 의무.

3.2 0 vs null vs N/A 데이터 상태 구분

RAW 데이터 상태 3종은 의미가 다릅니다. 혼동 시 외부 검증사 즉시 지적 가능.

상태의미대시보드 표시예시
0측정·관리하나 실적 없음검정 "0"법규 위반 건수 0건, 환경 부담금 0원
null해당 없음 또는 미입력회색 "—"측정 자료 미회신, 시설 부재로 측정 불가
N/A구조적 해당 없음 (NA 등록)회색 "─"가흥무역 = 무역법인이라 폐기물 측정 불가

N/A 판정 3단계

  1. 1단계 즉시 전환: 시설·인프라 부재, 사업 유형 불일치, 측정 대상 외
    • 예: 취저우스판 행74 지하수 → 지하수 취수 시설 부재
    • 예: 가흥무역 폐기물 지표 → 무역법인이라 제조 활동 부재
  2. 2단계 0 유지: 실적 미발생이 유의미한 지표
    • 예: 법규 위반 건수 0건 → 잘 관리되고 있음을 시사
    • 예: 환경 부담금 0원 → 비용 미발생을 명시
  3. 3단계 법인 확인: 동일 유형 법인 간 0/VALUE 차이가 있어 판단 유보
    • 예: A·B·C 법인 모두 동일 시설 보유, A·B는 측정값 있는데 C만 0
    • 이런 경우 C 법인에 측정 여부 추가 질의 필요

실수 방지 팁

"0이 아니라 null 아닌가요?" 같은 질의가 외부 검증 단계에서 들어올 수 있습니다. 의심스러운 경우 RAW 정정 + V/NA basis에 정확한 의미 명시.

5.1 법인 회신 엑셀 7단계 표준 절차

각 사업장에서 회신한 환경 정량 데이터 엑셀은 다음 7단계 절차에 따라 검수됩니다.

7단계 절차

  1. 1단계: 수령 확인
    • 회신 일자·발신자·파일 무결성 확인
    • 회신 누락 시 재요청 메일 작성
  2. 2단계: SSOT 직접 검증
    • 대시보드 직접 검증 (메모리 의존 금지)
    • 현재 등록 RAW·V·NA 상태 확인
    • 충돌·갭 사전 식별
  3. 3단계: 회신값 ↔ 대시보드 RAW 비교
    • 3개년 (FY23·FY24·FY25) 모든 셀 1:1 비교
    • 차이 발생 항목 카테고리별 분류 (불일치 6카테고리)
    • YoY 이상치 (±30% 초과) 자동 검출
  4. 4단계: 정합성 점검
    • 단위 일관성 (ton·m³·천 KRW 등)
    • 통화 일관성 (CURRENCY_MAP·OVERRIDE)
    • 0 vs null 구분 정확성
  5. 5단계: 추가 질의 작성
    • 법인 통합 메일 (한국어 + 현지어)
    • 위험도 C 사전 합의 (단순 질의)
  6. 6단계: 회신 수령 후 RAW 정정
    • 위험도 A 사전 합의 (RAW 변경)
    • 백업 + str_replace 또는 Node.js 처리
    • JS 파싱 검증 + V/NA 충돌 점검
  7. 7단계: V/NA 등록 + 매뉴얼 갱신
    • 다국어 ko/en/zh 1:1 번역
    • INDICATOR_MANUAL updated 필드 갱신
    • 외부 심사원 엑셀 동시 갱신

왜 이 7단계인가?

법인 회신 데이터를 무비판 적용하면 외부 검증 단계 즉시 지적 가능. 7단계 절차는 메모리 §1.12 정부 공식 문서 우선 + §1.12.1 SEMS 신뢰성 예외 + §5.1 SSOT 원칙을 모두 통합한 표준 작업 흐름입니다.

5.2 불일치 6카테고리 분류

법인 회신값과 대시보드 RAW 비교 시 차이 발생 항목은 다음 6카테고리로 분류·처리합니다.

카테고리설명처리 방향
1. 단위 차이ton vs kg, m³ vs L 등단위 환산 후 일치 확인
2. 통화 차이VND vs USD, CNY vs KRW 등CURRENCY_MAP·OVERRIDE 검토
3. 시점 차이회계연도 ≠ 측정 기간1.1~12.31 보고연도 기준 일관 적용
4. 분류 차이지정 vs 일반 폐기물 등한국 처분코드·중국 처분 분류 검증
5. 측정 방식 차이SEMS vs 汇总 vs 자체 측정정부 공식 문서 우선, SEMS 예외 적용

  2. 차이 ±30% 이내: SEMS 수용
  3. 차이 30% 초과: 사안별 검증 필요

5.8 폐기물 기말 미처리 재고 처리

폐기물은 발생 시점이 아닌 처리 완료 시점 기준 연도 분류가 원칙입니다.

왜?

발생 시점 기준 분류 시 기말 미처리 재고가 다음 회계연도로 이월되어 측정 누락 위험. 처리 완료 시점 기준이 GRI 306 권고.

처리 시점 분류 4종

  1. 당해 발생 + 당해 처리: 당해 보고 (정상)
  2. 당해 발생 + 익년 처리: 익년 보고 (이월)
  3. 전년 발생 + 당해 처리: 당해 보고 (전년 미처리분 포함)
  4. 당해 발생 + 미처리 (보고 시점): 다음 회계연도 처리 시 보고

네트웍스와이 일괄 정합 (메모리 #9)

외부 전문업체 네트웍스와이가 Scope 3 산정용 폐기물 데이터 제공 예정. 수령 시 대시보드 전수 비교 후 일괄 교체.

한국은행 ECOS cross-rate 환산 산식

동나이 인증비 외화 표시(USD·EUR 등)를 VND 환산 시 한국은행 ECOS 매매기준율 연평균 KRW cross-rate 적용:

VND 환산값 = (외화 금액 / KRW × 100) ÷ (100 VND / KRW)

예시 (FY25 USD → VND):
  USD/KRW 연평균 = 1,367.21
  VND/KRW 연평균 = 0.05642 (100 VND/KRW)
  
  USD 1,000 → VND 환산:
    (1,000 × 1,367.21) ÷ 0.05642 = 24,234,672 VND

본 정책은 향후 다른 통합 plant 사업장(예: 닝샤스판 1144+3741)에서 인증서 분리 발급 시 동일 적용. 외부 검증 답변 패키지에 표준 시나리오로 등록 (부록 E).

6.3 RAW 정정 절차

RAW 정정은 위험도 A 작업입니다. 다음 5단계 절차 준수.

  1. 백업: cp dashboard_v(N).html dashboard_v(N+1)_backup.html
  2. 대상 식별: 라벨 직접 검색 (idx 가정 금지)
  3. 정정 실행:
    • 소량 (1-2건): str_replace로 직접 수정
    • 대량 (10건+): Python + Node.js 객체 직렬화

    • HTML 외부 구조 5종 (DOCTYPE·html·body·script 쌍)

    • V/NA 충돌 점검

표준 정정 패턴 (Python + JSON 직렬화)

# RAW 객체 추출 (bracket-depth 정규식)
def find_obj_bounds(html, var_name):
    pattern = re.compile(rf'(?:const|let)\s+{var_name}\s*=\s*\{{')
    m = pattern.search(html)
    if not m: return None, None
    obj_start = m.end() - 1
    depth = 0
    for i in range(obj_start, len(html)):
        c = html[i]
        if c == '{': depth += 1
        elif c == '}':
            depth -= 1
            if depth == 0: return obj_start, i + 1
    return None, None

raw_s, raw_e = find_obj_bounds(html, 'RAW')
RAW = json.loads(html[raw_s:raw_e])

# 라벨로 직접 검색 (idx 가정 금지)
target_label = '환경 R&D·기술 투자'
ind = next((i for i in RAW['indicators'] if i['label'] == target_label), None)

# 정정
ind['data']['구미'] = [None, None, None]

# 직렬화 + 교체
new_raw_str = json.dumps(RAW, ensure_ascii=False, separators=(',', ':'))
html = html[:raw_s] + new_raw_str + html[raw_e:]

6.5 NA 등록 3단계

NOT_APPLICABLE 등록은 N/A 판정 3단계 절차에 따라 진행합니다.

1단계: 즉시 전환

다음 사유 발생 시 사업장 회신 대기 없이 즉시 NA 등록:

  • 시설·인프라 부재 (예: 취저우스판 행74 지하수)
  • 사업 유형 불일치 (예: 가흥무역 폐기물 지표)
  • 측정 대상 외 (예: 효성티앤씨 R&D 효성기술원 단독)

2단계: 0 유지

다음 경우는 NA가 아닌 0 유지:

  • 실적 미발생이 의미 있는 정보 (예: 법규 위반 0건 = 잘 관리됨)
  • 측정 결과가 명확히 0 (예: 환경 부담금 0원)

3단계: 법인 확인

판단 유보 시 법인 추가 질의:

  • 동일 유형 법인 간 0/VALUE 차이
  • 외부 검증사 지적 가능 항목
  • 증빙 자료 부재

// 5객체 검증 시 무결성 자동 확인
const subsSet = new Set(RAW.subs);
const domOverSet = new Set([...RAW.domestic, ...RAW.overseas]);
const inSubsNotInDomOver = [...subsSet].filter(s => !domOverSet.has(s));
const inDomOverNotInSubs = [...domOverSet].filter(s => !subsSet.has(s));
console.assert(inSubsNotInDomOver.length === 0, 'subs에 있으나 domestic·overseas 미포함:', inSubsNotInDomOver);
console.assert(inDomOverNotInSubs.length === 0, 'domestic·overseas에 있으나 subs 미포함:', inDomOverNotInSubs);

한자 사전 검증

주의: 사업장명 한자 사전 검증 필수

중국어 매핑 시 한국 사업장명을 음역(音译)이 아닌 의역(意译)으로 표기. 한자 표기는 본사 ESG경영팀이 사전 검증 후 등록.

  • 반포 = 盘浦 (반포동의 한자 표기)
  • 용연 = 龙渊 (용연동의 한자 표기)
  • 옥산 = 玉山 (옥산공장 소재지 한자 표기)

9.3 답변 패키지 표준 형식

외부 검증 사전 정리 답변 패키지는 다음 표준 형식으로 작성합니다.

답변 패키지 구성

  1. 질의 번호 + 사항
  2. 현재 상태 요약 (RAW 값 + V basis)
  3. 답변 방향 (정책 근거 + 증빙 출처)
  4. 증빙 자료 첨부 (스크린샷·PDF·CSV 등)
  5. 차후 조치 계획 (해당 시)

표준 답변 예시

예시: 닝샤 행75 용수 재활용 269,116% 폭증 답변

질의:
닝샤스판 행75 용수 재활용 FY24 → FY25 130,140,000 ton (269,116% 증가) 사유?
현재:
RAW: FY24 48,342 → FY25 130,140,000 ton (V 등록 대기)
답변:
C11 신증설로 인한 실제 폭증입니다. 폐쇄형 순환 냉각수 시스템 (closed-loop cooling water system)의 누계 순환량 계상 방식으로, C11 신규 라인 도입으로 새로운 대규모 순환 시스템이 추가되었습니다. 닝샤 측 증빙 자료 확보 완료, 차후 V 등록 예정.
증빙:
닝샤 자체 산출 자료「循环水量」 23·24·25 (메모리 §14)

예시: 한국 3공장 (구미·대구·울산) 매출 동일값 정책 답변

질의:
한국 사업장 행85 (친환경 매출) + 행86 (총 매출) RAW 값이 구미·대구·울산 모두 동일한 사유? 특히 울산 (NY·PET 공장)에 SP 사업부 매출이 포함되어 있는 것으로 보이는데?
현재:
한국 3공장 RAW FY25 1,076,482,640 천 KRW 동일 적용 (FY24 1,245,014,288 / FY23 1,179,602,786)
답변:
효성티앤씨 한국 본사는 섬유 사업부 매출이 사업장 단위로 회계 분리되지 않습니다. 본사 섬유 사업부 5개 (SP·NY·PET·T/C·기타)의 매출 합을 한국 3공장 (구미·대구·울산) 모두 동일값 적용하는 정책입니다. 외부 공시 시 한국 합계로 통합 보고하므로 사업장별 분리 보고 시 의미 손실 없음. 행85 친환경 매출은 SP 사업부 + 폴리에스터 사업부의 친환경 매출 합 (NY 미포함, 외부 공시명 "지속가능한 솔루션" = 리젠 + 리젠테크).
증빙:
경영기획팀 M_1-3 경영지수 12월누계 R10 소계 (24년·25년 자료 FY24 교차 검증 차이 0)

9.5 답변 시 INDICATOR_MANUAL·V basis 활용법

외부 검증사 답변 시 매뉴얼·V basis를 즉시 참조 자료로 활용합니다.

매뉴얼 활용 시나리오

  1. 외부 검증사 질의 수령
  2. 대시보드에서 해당 지표명 + ⓘ/📖 아이콘 클릭
  3. 모달 8필드 즉시 표시
  4. definition·include·exclude·decisionTree 답변에 활용
  5. 증빙 출처 (evidence) 첨부 자료 식별

V basis 활용 시나리오

  1. 특정 사업장 + 지표 V basis 마우스 오버
  2. 다국어 툴팁 즉시 표시
  3. basis 배열 답변 자료로 직접 인용
  4. 외부 검증사 언어에 맞춰 ko/en/zh 선택

답변 시간 절감 효과

매뉴얼·V basis 사전 작성 → 외부 검증 답변 시간 50% 이상 절감 가능. 메일·구두 답변 시 매뉴얼 모달 스크린샷 첨부 → 답변 신뢰성 향상.

9장 점검 체크리스트

  • 송부 전 10개 점검 사항을 안다
  • 외부 검증사 예상 질의 21선 답변 방향을 안다
  • 답변 패키지 표준 형식을 작성할 수 있다
  • 표기 금지 사항 5종을 외울 수 있다
  • 매뉴얼·V basis 활용 시나리오를 안다

10장. 버전 관리 + 백업

10.1 대시보드 버전 정책

핵심 원칙

대시보드 버전은 vNN 단순 정수 증가 정책 적용. 위험도 A·B 작업 1건당 +1 증가.

버전 표기 규칙

  • 형식: vNN (소문자 v + 정수, 점 없음)
  • 예: v1, v2, ..., v97, v98, v99, v100
  • 날짜 별도 표기: v98 (2026-05-03)
  • 버전 표기 위치: 헤더·파일명·표지 시트·메모리

버전업 시점

작업버전업
RAW 일괄 정정 (10건+)+1
VERIFIED·NA 일괄 등록 (10건+)+1
INDICATOR_MANUAL 일괄 갱신+1
버그 수정 (단순 표시 오류)+1
단순 조회·분석버전업 없음

10.3 산출물 백업 운영 원칙

대시보드·외부 심사원 엑셀·지침·인계 프롬프트 등 산출물은 별도 백업 저장소(클라우드 드라이브·공유 폴더·로컬 NAS 등)에 보관합니다. 구체적인 백업 도구·폴더 구조는 담당자 책임 하에 결정하며, 본 매뉴얼은 일반 운영 원칙만 안내합니다.

백업 운영 원칙 4종

  1. 발행 즉시 백업: 산출물 발행 시 즉시 백업 저장소에 동기화 (대시보드 vNN HTML + 외부 심사원 엑셀 vNN xlsx + 지침 vN MD + 인계 프롬프트 vNN MD)
  2. 버전별 보관: 직전 버전 덮어쓰기 금지, vNN 기준 별도 보관 (외부 검증 단계 변경 이력 추적 위해)
  3. 분류별 폴더 분리: 지침·대시보드·엑셀·인계 프롬프트·매뉴얼 등 5종 카테고리 폴더 분리 권고
  4. 접근 권한 관리: ESG경영팀 접근 권한 + 외부 검증사 임시 접근 권한 별도 운영

담당자별 도구 선택 자율

Google Drive·OneDrive·Dropbox·사내 NAS·GitHub·자체 백업 솔루션 등 담당자가 운영 중인 백업 도구를 활용합니다. 도구별 자동 백업 한도·동기화 절차·접근 제어 방식이 다르므로 본 매뉴얼은 도구 선택을 강제하지 않습니다.

10.4 산출 임시 저장 + 파일명 규칙

산출물은 작업 임시 폴더에서 생성한 후 백업 저장소로 이관합니다. 임시 폴더 경로는 작업 환경에 따라 다를 수 있습니다.

임시 저장 운영

  • 최신 vNN + 직전 v(N-1) + 직전 v(N-2) 보관 (3 generation 권고)
  • 차차전 v(N-3) 이전은 백업 저장소로 이관 후 임시 폴더 정리
  • 임시 작업 파일은 별도 작업 폴더 사용 (산출 폴더와 분리)

파일명 규칙 (필수)

  • 날짜 + 버전 포함 (예: 대시보드_2026-05-03_v98.html)
  • 한글 파일명 사용 가능
  • 공백·특수문자 (?·*·:) 사용 금지
  • 버전 번호는 정수 (v98·v99·v100) 점 표기 금지

10.6 수동 업로드 절차

  • 수동 업로드 표준 절차 6단계를 안다

def find_obj_bounds(html, var_name):
    """HTML에서 const/let 객체 정확 추출 (괄호 깊이 추적)"""
    pattern = re.compile(rf'(?:const|let)\s+{var_name}\s*=\s*\{{')
    m = pattern.search(html)
    if not m: return None, None
    obj_start = m.end() - 1
    depth = 0
    for i in range(obj_start, len(html)):
        c = html[i]
        if c == '{': depth += 1
        elif c == '}':
            depth -= 1
            if depth == 0: return obj_start, i + 1
    return None, None
